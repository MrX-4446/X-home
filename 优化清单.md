# 项目优化清单

记录待办的优化与新功能设想。按优先级排列，**现阶段大多不用急着做**，等规模上来或有精力时再逐项处理。

## 一、性能与存储（数据量大后再做）

> 存储底层已从 JSON 文件迁到 **SQLite**（`.local-storage/data.db`，键值单表，一类数据存一行 JSON 字符串，见 `lib/storage.js`）。这一步解决了「写文件崩溃损坏」和「备份麻烦」，但**读写仍是整块加载 / 整块覆盖**，性能特征与 JSON 相同。下面 1、3 项的性能问题依旧存在，需要「行结构化」才能根治（见第 6 项）。

### 1. AI 回复相关的全量读写隐患
**问题**：每回复一条消息，服务器都会：
- 多次全量读取 `memories`（`readStorage('memories')` 把整块读出、parse 成大数组）并对**每一条**记忆计算语义相似度（`lib/memory/surface.js`）。
- 更新记忆激活次数后，把**整个** memories 全量序列化写回（`server.local.js` 约 L419-431）。

记忆量在几千条内完全无感；到 1 万条以上开始有几百毫秒级开销；10 万条+ 才会明显拖慢。

**方案**（届时择一）：
- 把「每条消息全量写回记忆库」改成异步 / 批量写，不阻塞回复。
- 给记忆检索加简单预筛选，减少全量遍历打分。
- 把 memories 做成 SQLite **行结构化**表（一行一条记忆 + 索引），支持按需查询单条、走索引筛选，不再整块读写（见第 6 项「方案 B」）。

### 2. 书籍正文单独存储
**问题**：导入的整本书正文目前直接存进笔记 JSON，列表加载时会把大段正文一起读进内存。

**方案**：书籍原文单独存成 `book-{id}.txt`，笔记里只保留路径引用，列表加载不再读整本书。

### 3. 老记忆定期归档 / 合并
**问题**：记忆只增不减，长期会越来越大。

**方案**：复用现有日记 / 周记归档雏形，定期把老记忆压缩合并，控制记忆库体量。

## 二、记忆系统三大诉求（好查找 / 防丢 / 省 token）

以下三点是不同层面的事，别指望一个方案全解决。其中 **4（数据防丢）、5（省 token）已基本完成并移入《功能清单》**，此处只剩 4 的自动备份、6 的语义检索。

### 4. 数据防丢 —— 剩余：更省心的自动备份
> **一键导出/导入已完成并移入《功能清单》。** 此处仅保留尚未做的自动备份方案。

**可继续（更省心的自动备份，按需再做）**：
- **Git 私有仓库自动备份**：服务器定时 `git add .local-storage && commit && push` 到私有仓库，服务器没了数据仍在远端。
- **对象存储**：定时上传到阿里云 OSS / 腾讯云 COS / S3，按量付费很便宜，较专业。

### 6. 真正的语义检索（改动最大，效果最好，最后做）
**问题**：当前 `calculateTextSimilarity`（`lib/memory/core.js`）是 **TF 余弦相似度**，只按字/词**字面重叠**打分。同义不同字（「妈妈」vs「母亲」、「生日」vs「诞辰」）匹配不到，不是真正的语义理解。

**已做（方案 A：SQLite kv 单表）**：底层已迁到 SQLite，但仍是「一行存一整块 JSON」，`readStorage('memories')` 依然全量读、全量遍历打分。它带来了防写坏 + 好备份，**但没提升检索性能，也不等于语义检索**。

**要做大数据量语义检索需要「方案 B：行结构化 + 向量」**：
- **行结构化**：把 memories 从「一行一大块」改成「一行一条记忆」，带列（chat_id / content / importance / is_active…）和索引，检索时走索引只取需要的记忆，不再整块读入内存。
- **向量嵌入 embedding**：每条记忆存一个向量（调 embedding API 生成，存 BLOB 列），检索时算向量余弦相似度，才能匹配同义不同字。
  - 简单版：embedding 存 BLOB，仍在 JS 里算余弦（准但仍遍历）。
  - 专业版：装 `sqlite-vec` 扩展，让数据库直接做向量近邻查询，量大才真正快。

**升级路径（好消息）**：所有业务代码只认 `readStorage`/`writeStorage`，将来做方案 B 时可只在 `storage.js` 里给 memories 单独走行结构化表，其它数据继续用 kv 表，业务代码基本不动。

**触发条件 / 建议**：现在几百上千条记忆，字面匹配 + 全量遍历毫无感知，此时上 embedding 反而多花 API 钱、多写复杂度。**等记忆到上万条、或明显感到「想不起同义的事」时再做**。接 embedding API 有调用成本，数据量不大时收益不急。

## 三、部署与安全

### 7. 配置 AI 密钥加密（AI_CONFIG_SECRET）
**问题**：服务器未配置环境变量 `AI_CONFIG_SECRET`（`/api/ai-status` 返回 `ok:false`，「AI 密钥加密配置：未配置」）。未配置时，AI 服务商的 API Key 以**明文**存在服务器 JSON 文件里，安全性差。不影响聊天功能。

**方案**：
- 生成随机密钥：`openssl rand -hex 32`。
- 在 pm2 启动配置（ecosystem / `pm2 set`）里注入环境变量 `AI_CONFIG_SECRET`，然后重启后端。
- ⚠️ 若此前已用明文存过 AI 密钥，配好加密后需在设置里**重新填一次密钥**，让其以加密形式重新保存。
- 相关代码：`last/api/_lib/ai-providers.js`（无 secret 时会抛错拒绝保存）。

### 11. 纪念日 / 提醒（方案已定，待实现）
让「恋人 X」在生日、约定日期等纪念日主动发消息（倒计时预告 + 当天庆祝），比现在的随机想念更有温度。复用已有的 AI 主动消息 / 推送 / 上下文注入模式（`lib/memory/proactive.js`、`schedule.js`）。

**★ 已选定方案：公历+农历 / 提前3天+当天 / 独立模块；月历同步加农历+黄历。**

**农历/黄历库选型**：用 `lunar-javascript`（npm 纯 JS 包，无依赖，前后端通用）。提供农历日期、节气、生肖、干支、黄历宜忌/冲煞等，一个库同时满足「纪念日农历」和「月历显示农历+黄历」。前端 `first`、后端 `last` 各装一份。

**一、纪念日提醒（后端 `last/lib/memory/anniversaries.js`，独立模块，不塞进 schedule）**
- **数据层**：新增存储 key `anniversaries`，复用 SQLite kv 整存整取。单条结构：
  ```
  { id, title,              // "在一起纪念日" / "轩的生日"
    month, day,             // 该纪念日的月/日
    calendar: 'solar',      // 'solar' 公历 | 'lunar' 农历
    startYear: 2023,        // 可选，用于算"第几周年 / 几岁"
    leadDays: 3,            // 提前几天预告（默认3）
    note,
    lastNotified: '2026-07-08' }  // 防同一天重复推送
  ```
- **核心逻辑**（复用 `schedule.js` 的北京时间 / AI 生成 / 推送模式）：
  - `nextOccurrence(anni)`：算出「今年（或明年）该纪念日的公历日期」。农历的用 lunar-javascript 把「农历月日 → 今年对应公历日」。
  - `anniversaryTick()`：每天扫一次，对每条算 `diff = 距今天几天`：`diff===0` 当天庆祝语（带「第 N 周年」）；`0<diff<=leadDays` 倒计时预告（"还有 N 天就是…啦"）。用 `lastNotified` 保证一天只推一次；复用 `callAIProvider`(useHelperAI) + 恋人X人设 + `pushReminderToChat`。
  - `buildAnniversaryContext()`：把「未来 30 天内的纪念日 / 今天是不是纪念日」拼进系统提示，与现有 `buildScheduleContext`/`buildShiftContext` 并列注入，让平时聊天也能自然带出。
- **定时任务**：新增 `setupAnniversaryTask`（每 6 小时扫一次，靠 `lastNotified` 去重），在 `server.local.js` 启动时调用；新增路由 `GET/POST/DELETE /api/anniversary`。

**二、月历显示农历 + 黄历（前端 `CalendarPanel`）**
- **日期格子**：每格公历数字下方显示农历日 / 节气 / 节日（"初八""立秋""除夕"），用 `Lunar.fromDate()`。
- **点开某天**：当日面板显示黄历信息——农历全称（"丙午年六月初八"）、干支、生肖、宜 / 忌、冲煞、星期。
- 纪念日徽标复用现有排班徽标的绝对定位样式，避免与排班 / 日程点冲突。

**三、与第 15 项通知打通**：纪念日临近时预生成的话术，天然可作为极光推送 / 本地通知内容（做通知时直接复用）。

**实现顺序**：后端 anniversaries.js → 路由 / 上下文注入 → 前端农历黄历渲染 → 纪念日 UI。

### 14. 日历 / 日程系统 —— 剩余：系统日历
> **核心 + 排班表已完成并移入《功能清单》**（第十三章日历 / 日程；第十四章排班表）。此处仅保留尚未实现的读取手机系统日历。

**③+ 读取手机系统日历（已选定，让 AI 也了解系统里的安排）**
- 目的：AI 读到的日程 = App 自记（SQLite）**+** 手机系统日历事件（会议 / 生日 / 别的 App 写入的事件），合并后一起喂给恋人 X。
- 插件：`@ebarooni/capacitor-calendar` 或 `capacitor-calendar`，申请日历**读取**权限（`READ_CALENDAR`）。远程加载模式下插件仍可用，需装插件重新打包一次 APK。
- 读取方式：进 App / 进日历页时读**近 N 天**（今明~本周）系统事件，**不落库**，仅在构造 prompt 时临时合并注入，来源标注清楚（如 `[系统日历] 15:00 部门会议` vs `[我记的] 19:30 和轩看电影`）。
- ⚠️ **提醒归属**：系统日历事件的提醒由**系统原生**负责，我们**不**为它重复推送/发本地通知；只有 App 内自记日程才走恋人 X 人设提醒，避免重复打扰。
- ⚠️ **隐私**：只读近几天、只临时用于当次对话、不写库、不进记忆系统。
- ⚠️ **权限拒绝兜底**：用户不授权时自动降级为「只读 App 自己的 SQLite 日程」。

### 15. 手机系统通知（AI 消息弹到通知栏，方案已定，下次实现）
目标：让 AI 的消息（日程提醒 / 主动想念）像微信、QQ 那样弹到手机系统通知栏。已封装成 Capacitor APK，原生插件可用（远程加载模式下插件仍生效，但需重新打包一次 APK）。分两条路线，按需选做：

**路线 A：本地通知（推荐先做，配合第 14 项日历一起）**
- 插件：`@capacitor/local-notifications`。
- 原理：App 自己在设备上弹系统通知（真进通知栏，带图标 / 声音）。
- 最适合**日程提醒**：日程时间已知，可提前 `schedule` 好本地通知，**即使断网也能按时弹**。
- 前提：App 进程还活着（前台或后台未被杀）。被系统清理 / 划掉后不触发。
- 成本低：无第三方、无资质、改动小。

**路线 B：推送通知（真·微信/QQ 效果，App 关闭也能收）**
- 插件：**极光 JPush**（国内首选，有 Capacitor 插件）。不用 FCM——Google 服务在国产 ROM 上基本不可靠。
- 原理：服务器主动 push，系统级唤醒，**App 完全关闭也能弹**。
- 服务器改造：存设备 token → `lib/memory/proactive.js` 生成主动消息后，除写入 chat 外再调极光推送 API。
- 适合 **AI 随机主动消息**（想念）在 App 关闭时也能收到。
- 成本较高：需注册极光开发者、集成 SDK、服务器对接推送 API。
- **免费额度（自用绰绰有余）**：别名/RID 单推共享 **200 万条/天**、频率 600 次/分钟；离线消息缓存 5 条/最长存 3 天。⚠️ 有硬限制的是「广播接口」10 次/天，但推给自己一台设备走的是**别名单推**，不受此限。

**通知内容用 AI 人设生成（重点，两条路线都适用）**
- 通知栏显示的不是死模板，而是**「恋人 X」实时生成的话**：`reminderTick` 扫到日程后，把日程信息喂给 `callAIProvider` + 恋人 X 人设 prompt，生成一句口吻自然的提醒（如"轩~还有一小时就要看电影啦，记得早点准备哦♡"），这句话既写入聊天记录、又作为通知内容推送。与第 14 项第④步一致。

**关于「常驻后台」的结论（用户问过）**
- 光让 App 常驻后台**不够**：① 国产 ROM 激进省电会杀进程，需手动设「允许自启动 + 关闭电池优化 + 锁后台」；② 即使进程活着，后台 WebView 会被挂起、前端不会自己轮询，仍需原生侧「定时轮询 + 本地通知」代码才会弹。
- 靠常驻 + 轮询**不稳定且耗电**，体验远不如推送。要「App 关了也弹」只有路线 B 可靠。

**建议**：日程提醒 → 路线 A（省心）；AI 主动想念要在 App 关闭时也弹 → 路线 B（极光推送）。两条都需装插件并重新打包一次 APK，之后前端逻辑改动仍免打包。

**★ 已选定：A + B 结合方案（极光推送为主 + 本地通知兜底）**
核心思路：**后端是唯一的「人设消息生产者」，两个通道各司其职。**

通道分工：
| 消息类型 | 主通道 | 兜底通道 | 理由 |
|---|---|---|---|
| AI 随机主动想念 | 极光推送 | —— | 时间随机、App 常关，只有推送能唤醒 |
| 日程提醒 | 极光推送 | **本地通知** | 时间已知可提前 schedule，断网/服务器挂了也能准时弹 |

为什么日程要双保险：极光离线只缓存 5 条/存 3 天且依赖服务器+网络；而日程时间**已知**，可在新增日程时就把本地通知提前排好，**完全离线也能按时弹**。两条通道对同一日程要**去重**（本地通知带 `scheduleId`，收到推送时若本地已弹过则忽略）。

统一流程：
1. **设备注册**：App 启动拿极光 RegistrationID → `POST /api/push-token` 存入 SQLite（新 key `push_tokens`）。
2. **AI 生成人设话术**：`reminderTick`/`proactiveTick` 生成消息时统一用 `callAIProvider` + 恋人 X 人设出一句自然口吻的话。
3. **写库 + 双发**：这句话 ① 写入 chat（`proactive: true`）；② 调极光 API 推送；③ 若是日程，前端在新增/修改时用**同一句预生成话术**排好本地通知兜底。
4. **去重**：通知带 `scheduleId`，App 前台时优先走 chat 显示、避免重复弹。

落地顺序：先做**本地通知**（改动小、先让日程提醒能弹）→ 再接**极光推送**（覆盖 App 关闭场景）→ 最后做去重打磨。两者都需装插件并重新打包一次 APK，之后前端逻辑改动仍免打包。

## 备注

- 「快慢」的主要瓶颈是 AI 大模型本身的生成速度和发送的 token 量，存储读写只在数据量极大时才成为次要瓶颈。
- 存储已用 SQLite（`data.db`），存的是**硬盘**空间（不是运行内存），纯文字体量很难占满，不必担心存不下；要关注的是数据量变大后「整块读写」的性能（见第 1、6 项）。
- 数据库文件在 `.local-storage/`（已被 `.gitignore` 忽略）。备份直接用「应用检查」面板的导出功能，或复制整个 `data.db`（含 `-wal`/`-shm` 临时文件）。
