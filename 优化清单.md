# 项目优化清单

记录待办的优化与新功能设想。按优先级排列，**现阶段大多不用急着做**，等规模上来或有精力时再逐项处理。

---

## 一、性能与存储（数据量大后再做）

### 1. AI 回复相关的 JSON 全量读写隐患
**问题**：每回复一条消息，服务器都会：
- 多次全量读取 `memories.json` 并对**每一条**记忆计算语义相似度（`lib/memory/surface.js`）。
- 更新记忆激活次数后，把**整个** `memories.json` 全量序列化写回（`server.local.js` 约 L419-431）。

记忆量在几千条内完全无感；到 1 万条以上开始有几百毫秒级开销；10 万条+ 才会明显拖慢。

**方案**（届时择一）：
- 把「每条消息全量写回记忆库」改成异步 / 批量写，不阻塞回复。
- 给记忆检索加简单预筛选，减少全量遍历打分。
- 数据量真的很大时迁移到 SQLite（单文件数据库，支持按需查询单条 + 索引）。

### 2. 书籍正文单独存储
**问题**：导入的整本书正文目前直接存进笔记 JSON，列表加载时会把大段正文一起读进内存。

**方案**：书籍原文单独存成 `book-{id}.txt`，笔记里只保留路径引用，列表加载不再读整本书。

### 3. 老记忆定期归档 / 合并
**问题**：记忆只增不减，长期会越来越大。

**方案**：复用现有日记 / 周记归档雏形，定期把老记忆压缩合并，控制记忆库体量。

## 二、记忆系统三大诉求（好查找 / 防丢 / 省 token）

以下三点是不同层面的事，别指望一个方案全解决。推荐做的先后：4 → 5 → 6。

### 4. 数据防丢（最优先，成本最低）★
**问题**：所有数据都在云服务器的 `.local-storage/*.json`，**服务器到期 / 被删就全没了**，这是唯一「丢了无法挽回」的风险。与 JSON 格式无关，本质是**备份**问题（JSON 是纯文本，恰恰最好备份）。

**方案**（从轻到重，任选或组合）：
- **Git 私有仓库自动备份**（最推荐）：服务器定时 `git add .local-storage && commit && push` 到一个私有仓库，服务器没了数据仍在远端。几十行代码 + 一个定时任务。
- **定时导出下载**：后端加 `/api/export` 打包所有 JSON，定期手动下载存本地 / 网盘。简单但靠手动。
- **对象存储**：定时上传到阿里云 OSS / 腾讯云 COS / S3，按量付费很便宜，较专业。

### 5. AI 读取省 token（已做一半）
**现状**：`/api/chat` 已有相关度过滤（`RELEVANCE_THRESHOLD`）+ 只取 top 5，方向正确。

**可继续优化**：
- 每条记忆增加结构化 `summary` 摘要字段，发给 AI 时优先发摘要而非全文。
- 检索时先用关键词 / 标签粗筛，再打分，减少发送量。

### 6. 真正的语义检索（改动最大，效果最好，最后做）
**问题**：当前 `calculateTextSimilarity`（`lib/memory/core.js`）是 **TF 余弦相似度**，只按字/词**字面重叠**打分。同义不同字（「妈妈」vs「母亲」、「生日」vs「诞辰」）匹配不到，不是真正的语义理解。

**方案**（要真语义需要**向量嵌入 embedding**）：
- **保持 JSON + 加 embedding 字段**：每条记忆存一个向量（调 embedding API 生成），检索时算向量余弦相似度。不用换数据库。
- **换 SQLite**：单文件数据库，支持索引 / 按需查询单条，不用全量读写；备份也只需一个 `.db` 文件。是否换 SQLite 与要不要 embedding 是**两个独立决定**。
- **向量数据库**（sqlite-vec / Chroma 等）：量大时最优，但对单用户应用可能过度设计。
- 注意：接 embedding API 有调用成本，数据量不大时收益不急。

## 三、部署与安全

### 7. 配置 AI 密钥加密（AI_CONFIG_SECRET）
**问题**：服务器未配置环境变量 `AI_CONFIG_SECRET`（`/api/ai-status` 返回 `ok:false`，「AI 密钥加密配置：未配置」）。未配置时，AI 服务商的 API Key 以**明文**存在服务器 JSON 文件里，安全性差。不影响聊天功能。

**方案**：
- 生成随机密钥：`openssl rand -hex 32`。
- 在 pm2 启动配置（ecosystem / `pm2 set`）里注入环境变量 `AI_CONFIG_SECRET`，然后重启后端。
- ⚠️ 若此前已用明文存过 AI 密钥，配好加密后需在设置里**重新填一次密钥**，让其以加密形式重新保存。
- 相关代码：`last/api/_lib/ai-providers.js`（无 secret 时会抛错拒绝保存）。

### 8. 修复辅助 AI 配置（自动日记 / 周记整理）
**问题**：日志报 `[日记整理] 失败：辅助AI配置不可用：HELPER_AI_PROVIDER_ID=provider-...`。辅助 AI 指向的 provider id 在当前环境不存在，导致自动日记/周记整理失败。不影响正常聊天。

**方案**：在设置里把辅助 AI 的 provider 重新选一个当前可用的即可。

## 备注

- 「快慢」的主要瓶颈是 AI 大模型本身的生成速度和发送的 token 量，JSON 读写只在数据量极大时才成为次要瓶颈。
- 所有本地 JSON 存的是**硬盘**空间（不是运行内存），纯文字体量很难占满，不必担心存不下；要关注的是数据量变大后的读写性能。
