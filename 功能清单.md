# 功能清单

本项目是一个以「恋人 X」人设陪伴用户「轩」的 AI 陪伴聊天应用。
前端：React 18 + Vite 5；后端：Node.js 原生 HTTP（`last/server.local.js`，端口 8888）；数据存本地 SQLite（`.local-storage/data.db`，`lib/storage.js` 键值表，底层 better-sqlite3）；AI 走 OpenAI 兼容接口（火山方舟）。

以下为**已实现**的全部功能。

---

## 一、聊天对话

1. 发送文本消息并保存到后端；无会话时自动创建首个会话。（`App.jsx`）
2. AI 流式回复（SSE），边接收边打字机展示。（`App.jsx` + `lib/api.js` + 后端 `/api/chat/stream`）
3. 保留非流式对话接口作为回退。（后端 `/api/chat`）
4. 多轮上下文组装，仅取最近 20 条历史控制 token。（`App.jsx`）
5. 发送失败兜底：try/catch/finally 复位「输入中」状态、回填输入内容方便重发。（`App.jsx`）
6. 「正在输入」三点动画气泡。（`TypingIndicator.jsx` / `ChatArea.jsx`）
7. 消息已读/未读状态显示（用户消息按是否有 AI 回复、AI 消息按页面可见性判断）。（`ChatArea.jsx` + `Message.jsx`）
8. 智能滚动，浏览历史时显示「新消息 / 回到底部」浮动按钮。（`ChatArea.jsx`）
9. 工具调用过程可视化时间线（输入 / 输出 / 成功失败）。（`Message.jsx`）
10. 多行自适应输入框、回车发送、模型选择下拉。（`InputArea.jsx`）
11. 会话动态标题与在线状态点。（`ChatHeader.jsx`）

## 二、AI 主动消息与通知

12. 后端每 1 小时扫描所有会话，按条件（超 3 小时冷静期、当天未达 5 条上限、15% 概率、北京时间 7:00–23:00 活跃时段）生成「突然想起你」的主动消息。（`lib/memory/proactive.js`）
13. 前端每分钟轮询当前会话，发现 `proactive` 主动消息即刷新界面。（`App.jsx`）
14. 主动消息去重，避免历史消息重复提示。（`App.jsx`）
15. 首次进入请求浏览器通知授权；收到新主动消息弹出浏览器 Notification。（`App.jsx`）

## 三、AI 提供商配置

16. AI 接入健康状态检查与重新检查。（`AIConfigPanel.jsx` + 后端 `/api/ai-status`）
17. 预设 9 类服务商模板（火山 / OpenAI / Claude / 智谱 / 豆包 / 通义 / 零一 / DeepSeek / 自定义）自动填充端点。（`AIConfigPanel.jsx`）
18. 添加 / 启用停用 / 删除 / 测试连接 AI 提供商；API Key 明文存储、对外返回打码。（`AIConfigPanel.jsx` + 后端 `/api/ai-providers`）
19. 主 AI（对话）与辅助 AI（记忆压缩等后台任务，用低成本模型、不占主 AI Token）分离。（`lib/ai-provider.js`）
20. 已启用模型进入聊天页模型下拉供切换。（`App.jsx` + `InputArea.jsx`）

## 四、工具调用（工作间）

21. 工具列表按分类筛选、启用/停用、显示已启用数量。（`ToolConfigPanel.jsx` + 后端 `/api/tools`）
22. 新增自定义工具：普通工具 / AI 技能 / MCP 服务 / 本地手机 App（可配接口地址、启动命令、目标 App 与动作）。（`ToolConfigPanel.jsx`）
23. 聊天时把已启用工具以 OpenAI function calling 格式传给 AI，支持两轮工具调用（请求工具 → 执行 → 再总结）。（后端 `server.local.js` + `lib/tools.js`）
24. MCP 服务接通（HTTP 型）：前端添加 type=mcp 的工具并填 endpoint 后，后端 `buildToolDefinitions` 自动从该 MCP 服务器拉取 `tools/list`，把其全部工具展开成独立 function（命名空间 `mcp__<id>__<工具名>`）并注入真实 inputSchema 给 AI；AI 调用时按前缀路由回对应 MCP 服务器执行 `tools/call`。含 60s 工具清单缓存、SSE/JSON 双响应解析、15s 超时、结果 8KB 截断；MCP 不可用时注入占位 function 让 AI 感知并告知用户。（`lib/mcp-client.js` + `lib/tools.js` + `server.local.js`）

> stdio 型 MCP（command 启动命令）与手机本地 App 调用、Plan-Execute 循环见 [docs/agent-evolution-plan.md](./last/docs/agent-evolution-plan.md)。

内置工具：
- **网页搜索** `web_search`：博查 Web Search API，返回前 5 条带摘要结果。
- **打开网页** `open_webpage`：抓取指定 URL 正文，防 SSRF、15 秒超时、正文截断约 4000 字。
- **计算器** `calculator`：数学表达式求值（字符白名单校验）。
- **天气查询** `weather_query`：wttr.in 返回温度、天气、湿度、风速。
- **翻译** `translator`：MyMemory API，支持中英日韩法德西俄等互译。
- **系统时间** `system_time`：返回北京时间日期/时间/星期。
- **代码执行** `execute_code`：JavaScript 受限沙箱；Python 经 `/api/execute-code` 真实执行本机子进程。

## 五、记忆系统

25. 置顶（不衰减）、标记已解决（降权）、归档 / 恢复。（`MemoryPanel.jsx`）

84. 极光推送（JPush）：App 关闭也能收——App 启动上报 RegistrationID 存入 SQLite（key `push_tokens`），后端在日程提醒 / 主动想念 / 纪念日消息生成后调极光 REST API 单推。（`first/src/lib/push.js` + `last/lib/push.js` + `capacitor-plugin-jpushn`）

## 十七、记忆语义向量检索（embedding）

88. 记忆语义向量检索：新记忆持久化后自动算向量存入 `memory_vectors` 表（`onMemoryPersisted` 钩子）；检索时优先用向量余弦相似度作为语义分，关闭总开关 `MEMORY_EMBEDDING` 时自动回退关键词相似度（双保险，不影响主流程）。走 OpenAI 兼容 embedding 接口（阿里百炼 / 火山方舟通用），模型与接入经环境变量注入（`MEMORY_EMBEDDING` / `MEMORY_EMBEDDING_MODEL` / `EMBEDDING_AI_PROVIDER_ID`），不硬编码。（`lib/memory/embedding.js` + `lib/ai-provider.js` + `server.local.js`）
89. 两重「防串味」保护：①检索时只对与查询向量同维度的记忆算余弦，维度不符（换过模型留下的旧向量）跳过并打印告警；②存量回填检测到库里已有其它模型生成的向量时拒绝混存，须显式 `rebuild` 清空重算。（`lib/memory/embedding.js` `retrieveByVector` / `backfillAllVectors`）
90. 存量回填接口 `POST /api/memory/backfill-vectors`：给所有 active 记忆补算向量，幂等（跳过已算过的、可中断续跑）、批量（默认每批 10 条，对齐百炼 v3/v4 单次上限）。body 传 `{"rebuild":true}` 时先清空旧向量再全量重算（换模型后用）。（`server.local.js` + `lib/memory/embedding.js`）

# 使用说明

## 一、启用记忆向量检索（首次）

1. 在阿里百炼或火山方舟控制台开通 embedding 能力，拿到 **endpoint、模型名、API Key**。
   - 百炼推荐：模型 `text-embedding-v4`，endpoint `https://dashscope.aliyuncs.com/compatible-mode/v1/embeddings`（默认 1024 维）。
2. 前端「设置 → AI 接入」新增一个接入：填 embedding 的 **endpoint（务必以 `/embeddings` 结尾，前后不要有空格或反引号）+ 模型名 + Key**，并启用它，记下该接入的 id。
   - ⚠️ 注意：AI 接入里的「测试连接」按钮发的是聊天格式请求，对 embedding 接入天然不适用（会报错），不代表配置错误，请忽略。
3. 在后端 `.env` 配置（三项）：
   ```
   MEMORY_EMBEDDING=on
   MEMORY_EMBEDDING_MODEL=text-embedding-v4        # 必须与接入里填的模型名完全一致
   EMBEDDING_AI_PROVIDER_ID=<第 2 步记下的接入 id>
   ```
4. 重启后端（pm2：`pm2 restart chat-backend`）。
5. 回填存量记忆：`curl -X POST http://localhost:8888/api/memory/backfill-vectors -H "Content-Type: application/json" -d '{}'`，返回 `{"ok":true,"total":N,"done":N}` 即成功。

## 二、更换向量模型 / 维度

向量是「可再生的派生索引」，记忆原文始终在 `memories` 表，重算无风险。换模型步骤：

1. 前端修改该 embedding 接入的 **模型名 + endpoint**（若换厂商）。
2. 同步修改 `.env` 的 `MEMORY_EMBEDDING_MODEL`（务必与接入里的模型名一致），必要时改 `EMBEDDING_AI_PROVIDER_ID`。
3. 重启后端。
4. **清空重算**（关键，换模型必须带 `rebuild`）：
   ```
   curl -X POST http://localhost:8888/api/memory/backfill-vectors -H "Content-Type: application/json" -d '{"rebuild":true}'
   ```
   否则新旧维度/模型向量混存，检索会触发「维度不符」告警并跳过旧向量。

> 兜底：即使忘记 `rebuild`，保护①会跳过维度不符的旧向量、保护②会在回填时拒绝混存并提示，不会算错，只是新旧向量并存导致召回不全，重跑 `rebuild` 即可修复。

## 三、常见排查

- **回填 `done:0`**：逐一排查——① 接入 endpoint 是否以 `/embeddings` 结尾且无空格/反引号；② 模型名是否真实存在（如百炼是 `text-embedding-v4`，不要臆造名字）；③ 单批是否超厂商上限（百炼 v3/v4 为 10，已默认对齐）。可直接 `curl` 打厂商 embedding 端点验证 Key/模型/endpoint 是否可用。
- **看真实错误**：回填失败原因打印在后端日志（pm2：`pm2 logs chat-backend`），形如 `[向量回填] 批次 N 失败: <原因>`。
- **确认向量已入库**：查 `memory_vectors` 表条数与样本维度、模型是否符合预期。
- **模型名锁版本**：用带版本号的模型名（如 `text-embedding-v4`），避免使用会滚动更新的别名，防止维度或语义悄悄变化。
